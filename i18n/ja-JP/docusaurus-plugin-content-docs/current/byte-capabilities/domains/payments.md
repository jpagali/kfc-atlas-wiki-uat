# 決済と支払い手段

**何をするものか:** 顧客がどう支払うかを扱います。クレジットカード、バウチャー、分割支払い、ギフトカードを PCI 準拠かつ不正耐性のある形で処理します。

**なぜ重要か:** 決済は売上確定前の最後の障壁です。決済失敗や不十分な決済体験は、そのまま注文完了率の低下につながります。また、この領域はプラットフォーム内でも最もセキュリティ感度が高い部分です。

:::caution Byte Helium はカード情報を扱わない
カード情報は PSP SDK が直接取得します。PAN（カード番号）が Byte Helium を通過したり、Atlas に保存されたりすることはありません。Byte Helium が扱うのはトークンだけです。決済設定（PSP 認証情報、ルーティングルール、追加料金ポリシー）は Byte Portal と PSP Vault 側にあります。
:::

## 機能テーブル

| 機能 | 解決する課題 | 何をするか | 仕組み | 依存先 | 前提条件 | 制約 | 影響を受ける指標 |
|---|---|---|---|---|---|---|---|
| **カードで支払う** | 顧客がデビット / クレジットカードを使いたい | カードをトークン化し、PSP 経由で決済を処理する | PSP SDK がカード情報を取得し（Byte Helium に PAN は入らない）、トークンを Checkout Orchestrator へ送る。必要に応じて 3DS 認証を行う | PSP / Vault、Checkout Orchestrator | 有効なチェックアウトセッション | 3DS により操作負荷が増える場合あり。失敗時は再試行または別支払い手段へ切替 | 決済成功率 |
| **保存済みカードで支払う** | 顧客が保存済みカードを再利用したい | 保存カード一覧を表示し、選択カードで支払えるようにする | PSP Vault から保存済みトークンを取得し、選択トークンを payment intent に使用する | PSP / Vault | 顧客がサインイン済みで、カードが以前に保存されている | カードが期限切れでないこと。Vault 管理は PSP 側で、Byte Helium ではない | 保存カード利用率 |
| **保存済みカードを削除する** | 顧客が保存カードを消したい | 顧客の保存済み支払い方法からカードを削除する | Byte Helium 経由で PSP Vault へ削除リクエストを送る | PSP / Vault | 顧客がサインイン済み | 削除は取り消せない | Vault 管理利用率 |
| **バウチャーで支払う** | 顧客がデジタルバウチャーを持っている | バウチャーコードを全額または一部支払いに充てる | Voucher service でコードを検証し、注文の支払い手段として値を付与する | Voucher service、Checkout Orchestrator | 有効なバウチャーコードで、期限切れ / 使用済みでない | バウチャールールは市場ごとに異なる。検証はサーバー側。二重利用は 409 で防止 | バウチャー利用率 |
| **分割支払い（カード + バウチャー）** | バウチャーで全額を賄えない | バウチャー適用後の残額をカードで支払えるようにする | 先にバウチャーを適用し、残額をカード payment intent で処理する | Voucher service、PSP、Checkout Orchestrator | 有効なバウチャーと有効なチェックアウトセッション | 支払い順序は Orchestrator が管理 | 分割支払い率 |
| **バウチャーを購入する** | 顧客がデジタルギフトを購入したい | カードでバウチャーを購入できるようにする | PSP で決済し、Voucher service がバウチャーを発行してメール / SMS で配信する | Voucher service、PSP | 有効な決済手段 | バウチャー購入フローは市場ごとの有効化が必要で、通常注文とは別導線 | バウチャー販売数 |
| **ギフトカード情報を再送する** | 顧客がギフトカードメールをなくした | 登録済み連絡先へギフトカード情報を再送する | Gift Card service へレート制限つき再送 API を呼び、設定済みチャネルで送信する | Gift Card service、Messaging | 顧客がサインイン済みでギフトカードを購入済み | 悪用防止のためカード ID 単位で再送制限あり | 再送依頼率 |

## 技術ソース

<details>
<summary>📎 技術ソース: カード決済 / 分割支払い</summary>

- **FRD 参照:** FRD-HEL-016, FRD-HEL-017, FRD-HEL-018
- **TRD ドメイン:** Payments & Tenders
- **主要インターフェース / API:** Card Tokenise / Pay（PSP SDK）、Voucher Apply / Redeem、Submit Payment Intent（Orchestrator）
- **データ契約:** PaymentIntent（amount、currency、tenders[]）、TenderCard（token、scheme、3dsStatus - ログに PAN は含めない）
- **要点:**
  - PCI-DSS 準拠。カード情報取得は SDK が担当し、PAN は Byte Helium やログに保存されない
  - 3DS をサポートし、結果はテレメトリで追跡
  - 決済認証 / 売上確定の成功目標は 99.5% 以上
  - バウチャーの二重利用は冪等性により防止し、409 時は元トークンを返す
  - Payment API は加算的変更のみ。180 日の非推奨期間あり（プラットフォーム内で最長）

</details>
