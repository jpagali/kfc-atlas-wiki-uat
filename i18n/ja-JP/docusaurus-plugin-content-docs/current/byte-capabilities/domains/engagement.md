# エンゲージメントとメッセージング

**何をするものか:** アプリ内メッセージを配信し、プッシュ通知設定を管理することで、CRM キャンペーンと顧客のアプリ内体験をつなぎます。

**なぜ重要か:** アプリ内メッセージとプッシュ通知は、注文間の主要リテンション導線です。設定管理が不十分だと信頼を損ない、オプトアウトにつながります。

---

## 機能テーブル

| 機能 | 解決する課題 | 何をするか | 仕組み | 依存先 | 前提条件 | 制約 | 影響を受ける指標 |
|---|---|---|---|---|---|---|---|
| **アプリ内メッセージ受信箱** | ブランドがアプリ内で顧客と接点を持ちたい | パーソナライズされたメッセージ、プロモーション、更新情報をフィード表示する | Engagement platform がページング付きメッセージフィードを返し、各メッセージはアプリ内ディープリンク可能 | Engagement platform、CRM | パーソナライズ表示はサインイン済み、グローバル表示は匿名でも可 | フィード鮮度は 60 秒以下。ディープリンク先が有効である必要あり | 受信箱開封率、メッセージクリック率 |
| **プッシュ通知設定** | 顧客が受け取る通知を制御したい | チャネルごとにプッシュ通知を ON / OFF できる | CRM で設定を読み書きし、変更時に端末トークンを同期する | CRM / Push provider、Push device token | 顧客がサインイン済みで、端末の通知許可がある | 端末単位とユーザー単位の両スコープあり。CRM 同期が必要 | プッシュ許諾率、オプトアウト率 |
| **メールマーケティング設定** | 顧客がメール配信設定を管理したい | メールマーケティングの登録 / 解除を行う | CRM Prefs API を更新し、監査履歴を残す | CRM | 顧客がサインイン済み | 監査履歴は必須。必要に応じて CMP 同期 | メール許諾率 |

---

## 技術ソース

<details>
<summary>📎 技術ソース: アプリ内受信箱 / プッシュ設定</summary>

- **FRD 参照:** FRD-HEL-051, FRD-HEL-065, FRD-HEL-066
- **TRD ドメイン:** エンゲージメントとメッセージング
- **主要インターフェース / API:** In-App Feed、Message Acknowledge、Push Prefs Get / Update
- **データ契約:** InAppMessage（id、title、body、deeplink、expiry）、PushPrefs（enabled、ts、source）
- **要点:**
  - フィード鮮度目標は 60 秒以下
  - 設定変更の監査履歴が必須
  - フィードが古い場合は pull-to-refresh を発火
  - 設定更新時に push token を同期
  - Engagement API は加算的変更のみ、120 日の非推奨期間

</details>
