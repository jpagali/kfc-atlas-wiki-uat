# チェックアウト

**何をするものか:** 顧客をカートから注文確定まで導き、配送情報を収集し、必要な検証を行い、注文意図を送信します。

**なぜ重要か:** チェックアウトは最も重要なステップです。ここで失敗すると売上を失います。ゲストユーザー、デリバリーと受取の違い、送信エラーへの耐性をすべて考慮する必要があります。

---

## 機能テーブル

| 機能 | 解決する課題 | 何をするか | 仕組み | 依存先 | 前提条件 | 制約 | 影響を受ける指標 |
|---|---|---|---|---|---|---|---|
| **ゲストチェックアウト** | すべての顧客が会員登録したいわけではない | サインインせずに注文できるようにする | セッションベースでチェックアウトを行い、OTP メールまたは匿名セッションを使う。カートはセッションに紐づく | Checkout Orchestrator、Address service | 商品入りの有効なカート | ゲストセッションには注文履歴もロイヤルティもない。住所保存もされない | ゲストチェックアウト転換率 |
| **チェックアウト - デリバリー** | 顧客が自宅や指定住所へ配達してほしい | 配達先住所を受け取り、手数料を表示し、注文を送信する | 住所を検証し、配達手数料を見積もり、配送情報つきで注文送信する | Checkout Orchestrator、Address service、Geo service | デリバリーモード設定済み。配達対象店舗が有効 | 住所が配達範囲内である必要あり。手数料はバックエンド計算で Byte Helium の責務ではない | デリバリーチェックアウト率 |
| **チェックアウト - 受取** | 顧客が店舗受取を希望している | 店舗選択と受取時間を含む簡易チェックアウトを提供する | 店舗確定後、住所不要で注文送信する | Checkout Orchestrator | 受取モード設定済み。店舗が営業中 | 店舗が営業状態である必要あり。スロットリング方針により注文が制限される場合がある | 受取チェックアウト率 |
| **デリバリーチェックアウトのバリアント** | 市場やケースごとにチェックアウトの流れが異なる | 市場別のステップ構成をサポートする | Orchestrator がコンテキストに応じたセッションを管理し、手順を出し分ける | Checkout Orchestrator | 市場バリアントが設定済み | バリアントは市場チームが設定する必要があり、Byte Helium から自己設定はできない | バリアント別完了率 |
| **再注文** | 顧客が過去注文をもう一度したい | 過去注文の商品をカートへ入れ直す | 過去注文の詳細を取得し、商品をカートへ再投入してチェックアウトを再開する | Orders service、Cart Service | 顧客がサインイン済みで過去注文が存在する | 現在のメニューにない商品は警告つきで除外される | 再注文率 |

## 技術ソース

<details>
<summary>📎 技術ソース: ゲストチェックアウト / デリバリーチェックアウト</summary>

- **FRD 参照:** FRD-HEL-012, FRD-HEL-013, FRD-HEL-014, FRD-HEL-015
- **TRD ドメイン:** Checkout（Core）
- **主要インターフェース / API:** Start / Resume Checkout、Address Create / Validate、Fees & Taxes Quote、Submit Order
- **データ契約:** CheckoutSession（id、cartRef、mode、steps）、OrderIntent（totals、tenders、deliveryInfo - 冪等性キー付き）
- **要点:**
  - チェックアウト送信の p99 成功率目標は 99.5% 以上
  - 冪等性キーにより再送時の重複注文を防止し、409 リプレイ時は元の orderRef を返す
  - 手数料と税はバックエンドで計算し、Byte Helium は表示のみ担当
  - Orchestrator API は加算的変更のみ、120 日の非推奨期間あり
  - すべての送信イベントで correlation ID を追跡

</details>
